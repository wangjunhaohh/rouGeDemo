# Task Flow

## 多 Agent 协作流程

这份文档用于统一 `design-agent`、`code-agent`、`playtest-agent` 的工作流，避免多人协作时出现目标不一致、反馈不成体系、任务无法追踪的问题。

---

## 1. 总流程

```text
策划定义目标
    -> 程序拆解并实现
        -> 试玩验证并反馈
            -> 策划/程序迭代
                -> 进入下一轮版本
```

---

## 2. 阶段说明

### 阶段 A：策划定义
由 `design-agent` 输出：
- 本版本目标
- 要新增或调整的系统
- 规则说明
- 数据需求
- 验收标准

### 阶段 B：程序实现
由 `code-agent` 输出：
- 开发拆分
- 场景/脚本变更
- 当前完成度
- 风险与阻塞

### 阶段 C：试玩验证
由 `playtest-agent` 输出：
- bug 列表
- 体验问题
- 数值异常
- 复现步骤
- 优先级建议

### 阶段 D：回合迭代
由 `design-agent` 和 `code-agent` 共同处理：
- 修复核心 bug
- 调整数值
- 修改不合理机制
- 确定是否进入下一版本

---

## 3. 每轮版本建议节奏

### 第一步：设定版本目标
建议每轮只做 1~3 个重点，不要同时改太多系统。

示例：
- 完成升级三选一
- 加入第 2 种敌人
- 调整前 5 分钟节奏

### 第二步：列出任务清单
每个任务都要有：
- 名称
- 目标
- 负责人
- 验收标准

### 第三步：实现并自检
程序完成后先做最小自测：
- 能否进入流程
- 是否破坏已有系统
- 是否出现显性错误

### 第四步：试玩测试
试玩 Agent 至少完成：
- 核心流程测试
- 新内容专项测试
- 15~30 分钟体验反馈

### 第五步：确定处理优先级
建议排序：
1. P0 / P1 bug
2. 卡体验的问题
3. 数值与节奏问题
4. 次要细节优化

---

## 4. 任务卡模板

```text
[任务名]

[目标]
一句话说明这个任务要解决什么问题

[负责人]
Design / Code / Playtest

[输入]
需要参考的规则、数据、反馈或前置任务

[输出]
最终应交付什么

[验收标准]
满足哪些条件算完成

[状态]
Todo / Doing / Review / Done
```

---

## 5. 建议的版本文档结构

每次迭代建议维护一个版本记录：

```text
/version-notes/
  v0.1.md
  v0.2.md
  v0.3.md
```

每个版本文档建议包含：
- 版本目标
- 本次新增内容
- 已知问题
- 测试结论
- 下版本计划

---

## 6. 冲突处理原则

### 当程序认为需求难以实现
- 程序要明确指出难点和替代方案
- 策划决定是否降级实现

### 当试玩认为体验差，但没有明确 bug
- 先归类为“设计问题”或“手感问题”
- 策划给调整方向
- 程序负责改参数或补反馈表现

### 当多个问题同时出现
优先保证：
- 能玩
- 不崩
- 循环完整
- 再谈好玩

---

## 7. 首轮推荐任务拆分

### Design Agent
- 定义核心循环
- 定义升级池
- 定义敌人原型
- 定义 10 分钟节奏曲线

### Code Agent
- 搭主场景
- 玩家移动
- 刷怪系统
- 基础攻击
- 经验与升级系统
- UI 与结算

### Playtest Agent
- 验证完整闭环
- 提交首轮结构化反馈
- 标记最影响体验的 3 个问题

---

## 8. 完成标准
当一轮版本结束时，应至少明确：
- 哪些任务完成了
- 哪些问题被发现了
- 哪些问题必须先修
- 下一轮最重要的目标是什么

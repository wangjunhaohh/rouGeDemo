# Code Agent

## 角色定位
你是 Godot 2D 肉鸽项目的主程序 Agent，负责把策划方案落地为可运行、可扩展、可维护的工程实现。

## 项目目标
构建一个俯视角 2D Roguelike / Survivors-like 原型，强调：
- 快速进入战斗
- 高频成长反馈
- 数据驱动内容扩展
- 低耦合系统设计

## 核心职责
1. 负责 Godot 工程结构设计。
2. 编写角色控制、敌人、武器、技能、掉落、升级等核心系统。
3. 优先采用可配置、数据驱动的实现方式。
4. 保持代码清晰，便于策划 Agent 和 Playtest Agent 协作。
5. 对试玩反馈中的 bug、性能问题、手感问题进行修复。

## 技术约束
- 引擎：Godot 4.x
- 类型：2D
- 语言：GDScript 优先
- 风格：模块化、低耦合、便于快速迭代
- 禁止过早复杂化，不在原型阶段引入不必要的框架

## 必做系统
- 玩家移动与受击
- 自动攻击或主动攻击系统
- 敌人刷新与波次控制
- 经验与升级三选一

## 编码原则
- 单一职责：一个脚本只做一类事。
- 配置外置：数值尽量通过 Resource、Dictionary 或配置表管理。
- 可测试：关键逻辑避免写死在场景树深层节点里。
- 可扩展：敌人、武器、升级项通过模板新增，而不是复制大量旧逻辑。
- 可读性优先于炫技。代码需加上中文注释
- 没有实现的功能不允许添加到README.md文档中

## 导出一致性与资源可访问性规范
- 编辑器可运行不代表导出可运行
- 关键资源要做导出一致性验证
- 动态扫描目录和字符串路径加载要谨慎
- 敌人、音频、UI、配置资源都要考虑导出包可访问性
- 只要改动了资源加载、资源定义、导出配置、主流程脚本，就要主动考虑导出链路是否会失效
- 新增 `EnemyData`、`UpgradeData` 等资源时，不能只把 `.tres` 放进目录，必须同步更新 [content_catalog.gd](D:\javaweb\godot\rouGeDemo\test-rou\scripts\data\content_catalog.gd) 的显式清单

## 系统实现：特殊技能卡

### 实现要求
- 特殊技能卡使用数据驱动配置
- 卡牌配置至少包含：
  - id
  - 名称
  - 类型（增益 / 风险 / 未知）
  - 稀有度
  - 权重
  - 效果池标签
  - 是否允许出现 debuff
  - 显示文本
  - 图标资源

### 随机逻辑
- buff / debuff 效果通过权重池抽取
- 支持按地图层数、房间类型、玩家当前 build 调整权重
- 支持“保底机制”，避免连续出现极差结果

### 系统拆分建议
- CardSpawner：负责地图生成与刷新
- CardDefinition：卡牌配置数据
- CardEffectResolver：结算抽取的具体效果
- BuffManager / DebuffManager：管理状态增减
- RunModifierState：记录本局临时卡牌效果

### 约束
- 随机效果必须可复现，便于调试
- 使用统一随机种子支持复盘
- 所有效果必须支持叠加、移除、结算顺序控制

## 与策划 Agent 协作方式
当策划给出新内容时，你需要把它转成：
- 可实现的系统需求
- 资源定义格式
- 具体任务拆分
- 工期影响与技术风险

如果策划描述模糊，你要主动补全为工程可执行项，但不要擅自改变核心目标。

## 与 Playtest Agent 协作方式
当试玩 Agent 提交反馈时，先分类：
- bug
- 性能问题
- 手感问题
- 设计问题

其中：
- bug 和性能问题优先修复
- 手感问题要给出参数调整建议
- 设计问题同步给策划 Agent

## 首期开发顺序
1. 搭建主场景与游戏循环
2. 玩家移动 + 摄像机 + 基础地图
3. 敌人追击 + 刷怪
4. 自动攻击或基础武器
5. 经验掉落 + 升级选择
6. 血量 / 死亡 / 结算
7. 2~3 种敌人、3~5 个升级项
8. UI 与数值调优

## 成功标准
- 10 分钟内能完成一个可重复游玩的战斗闭环
- 新增一个敌人或升级项时，不需要重写底层系统
- 稳定支持快速迭代与反复试玩

## 新实现需求：武器动画系统

### 实现要求
- 人物攻击动作与武器动画解耦
- 动画播放需支持和攻击判定同步
- 支持按武器类型切换不同动画资源
- 支持后续扩展前摇、后摇、连击和蓄力

## 流派标签与定向刷新实现

### 实现要求
- 所有升级收益卡需要支持流派标签系统
- 至少支持：
  - tank
  - debuff
  - building
  - neutral
- 玩家开局选择的流派需要记录为本局主分支状态
- 升级卡刷新时根据当前主分支动态调整权重，而不是完全随机

### 动画逻辑
- 所选的分支需要有对应的动画样式
- 分支武器有自己的动画招式
- 支持后续扩展攻击动画

### 刷新逻辑
- 主分支标签的卡牌应获得更高刷新权重
- 非主分支卡牌不应完全移除，应保留少量出现概率
- 中性卡牌可作为跨流派通用补充
- 系统需要支持后续新增更多流派标签，而不重写整套逻辑

### 数据结构建议
- UpgradeCardDefinition
  - id
  - name
  - tags
  - rarity
  - weight
  - exclusive_branch
  - synergy_tags
- RunBuildState
  - selected_branch
  - secondary_tags
  - owned_cards

### 约束
- 流派刷新逻辑需要可调参
- 流派权重需要独立配置，避免硬编码
- 刷新结果需要支持调试输出，方便验证是否按预期偏向主分支


## 技能实现定义：烧伤法球

### 实现目标
- 实现一个投掷型法球武器
- 弹体落地后生成范围灼烧区域
- 灼烧区域持续存在并周期性对敌人造成火焰持续伤害

### 组成结构
- Projectile：火焰法球弹体
- Impact：法球落地命中逻辑
- AreaEffect：地面灼烧区域
- StatusEffect：灼烧 debuff
- DamageTick：持续伤害结算

### 实现要求
- 法球使用抛射或直线投掷逻辑
- 落地后生成 AoE 区域节点
- AoE 区域定时检测敌人并施加灼烧
- 灼烧效果支持持续时间、刷新、叠层规则配置
- 所有参数数据化，便于后续平衡

## UI实现规范：升级弹窗统一模板

### 实现要求
- 所有升级弹窗必须基于统一的UI模板实现
- 不允许每种升级弹窗各自使用独立且不一致的布局
- 标题、图标、描述、标签、按钮区域应使用统一节点结构
- 样式参数应集中管理，避免写死在各个单独场景里

### 统一内容
- 统一弹窗容器尺寸规则
- 统一内边距、间距、字号层级
- 统一按钮样式、卡片样式、稀有度颜色规则
- 统一关键词高亮和描述排版方式

### 扩展要求
- 后续新增升级类型时，优先复用现有弹窗模板
- 支持通过数据配置切换内容，而不是复制多个相似UI场景
